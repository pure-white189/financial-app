### 第一战：攻克分布式系统的经典难题（云同步架构重构）

这是你目前系统里最“致命”的硬伤，也是面试中最能体现技术深度的部分。

*   **当前问题**：你的表（`expenses`, `loans` 等）用了 `id (auto)` 自增主键。在多设备同步时，两台手机离线记账都会生成 `id=1`，一联网同步就会发生数据灾难。
*   **重构方案**：
    1.  **废弃自增 ID**：让 Copilot 帮你写一个 Room Migration（比如 v15 -> v16），把所有涉及同步的表的主键类型从 `Int (AutoGenerate)` 改为 `String`。
    2.  **引入 UUID**：在客户端生成每条记录时，使用 `java.util.UUID.randomUUID().toString()` 作为主键。这样无论哪台手机离线生成的记录，它的 ID 都是全宇宙唯一的，永远不会冲突。
    3.  **时间戳改造**：在 `SyncRepository.kt` 中，往 Firestore 写数据时，不要用客户端手机的时间，改为写入 `FieldValue.serverTimestamp()`。这样冲突解决（LWW）就完全依赖服务器的绝对时间，防止用户手机时间错乱。
*   **面试加分话术**：“我在做多设备同步时，遇到了分布式系统经典的 ID 冲突和时钟漂移问题，后来我通过重构底层数据库，引入 UUID 主键和后端时间戳解决了这个问题。”

### 第二战：建立正确的安全观（永远不要相信客户端）

这是所有商业 App 开发的第一铁律。

*   **当前问题**：你的代币（Token）体系，加减逻辑写在 Android 端，并存入了本地 Room 数据库。
*   **重构方案**：
    1.  **剥夺本地的“记账权”**：客户端的 `token_transactions` 表只能作为“只读缓存”。
    2.  **后端接管资产**：在 FastAPI 后端现有的 `subscriptions.db` 中维护一个真实的用户余额表。
    3.  **接口改造**：把“签到”、“完成成就”这些动作改成调用后端 API：`POST /check-in`。后端验证今天是否真的没签到过，验证通过后在后端数据库给用户 +1 代币。客户端只负责展示后端返回的余额。

### 第三战：后端微服务可用性升级

从你的开发记录看，你的 Python 后端主要起到了中转和隐藏 API Key 的作用，但它的健壮性还不够。

*   **1. 拯救限流器（告别一重启就清零）**
    *   **重构方案**：你目前在 `main.py` 里的 AI 调用限速大概率是用 Python 的内存字典存的。让 Claude 帮你改写一下：把每天/每月的调用次数（`parse_count`, `analyze_count`）持久化写入你已经建好的 `subscriptions.db` 里，或者用 SQLite 写一个小表，记录 `(uid, date, count)`。这样即使 Azure 服务器重启，用户的配额也不会错乱。
*   **2. 拯救金融数据源（防封禁保护）**
    *   **当前问题**：每次用户查汇率/股票，后端就调一次 `yfinance` 去爬 Yahoo。并发一高，你的 Azure IP 必被封。
    *   **重构方案（加缓存）**：在 FastAPI 中引入基于时间的缓存（可以用 Python 自带的 `cachetools` 的 `TTLCache`）。比如汇率数据，设定缓存过期时间为 1 小时。
    *   **逻辑**：一小时内哪怕有 1000 个用户请求 `/exchange-rate`，后端也只在第一次去访问 Yahoo，后续 999 次直接从内存返回。这叫**“降低对下游的穿透率”**，是非常高级的后端工程思想。

### 一、 绝对致命的安全红线：裸奔的 HTTP 通信
*   **记录出处**：“后端部署”与“已知问题与限制：后端使用 HTTP（无 HTTPS），暂无域名”；以及“管理员生成激活码需 `X-Admin-Key` 请求头”。
*   **隐患分析**：在内网测试用 HTTP 没问题。但在公网（Azure IP 暴露在外），**所有数据都是明文传输的**。
    *   你的客户端每次请求都带着 `Authorization: Bearer <Firebase ID Token>`，任何人只要在同一个公共 Wi-Fi 下抓包，就能瞬间盗走这个 Token，伪装成该用户读取他的所有财务数据。
    *   你辛苦设计的 `ADMIN_KEY=666666`，在 HTTP 下就如同写在明信片上一样，黑客抓包看到后，可以无限调用 `/admin/generate-codes` 生成 Pro 激活码。
*   **修复建议**：这是上线前**必须**解决的问题。由于你是学生，不想花钱买域名和证书，你可以：
    1.  **零成本方案**：去申请一个免费的二级域名（如 DuckDNS）。
    2.  利用 **Certbot (Let's Encrypt)** 在你的 Ubuntu 虚拟机上给 Nginx 免费配置一张 SSL 证书。这只需几行 Linux 命令，就能把 HTTP 升级为 HTTPS。

### 二、 逻辑炸弹：Firestore 软删除（Ghost Data）带来的高昂代价
*   **记录出处**：云端数据同步 -> “软删除：删除操作先标记 `isDeleted = true`，同步后云端同步再清除本地记录”。
*   **隐患分析**：这就是典型的“幽灵数据（Tombstone）吞噬存储”陷阱。
    *   当你在手机上删除一条账单，同步到云端后，云端变成了 `isDeleted = true`。然后你的手机清除了这条本地数据。
    *   **问题 1**：由于云端永远保留着这条标记为删除的记录，随着用户不断记账、删除，云端的垃圾数据会无限增长。
    *   **问题 2（极其费钱）**：Firebase Firestore 的计费逻辑是**“按文档读取次数（Document Reads）”**收费的。未来哪怕这台手机清空数据重新全量拉取，它也会把那成千上万条 `isDeleted = true` 的幽灵账单全下载一遍。如果用户基数大了，这会导致你的 Firebase 免费额度瞬间被刷爆。
*   **修复建议**：
    *   **引入 TTL（Time-To-Live）机制**：利用 Firestore 自带的 TTL 功能，设置让系统自动清理 `updatedAt` 超过 30 天且 `isDeleted == true` 的垃圾记录。客户端在这 30 天内有足够的时间完成多设备同步。

### 三、 竞态条件与 UI 穿透（并发漏洞）
*   **记录出处**：代币兑换后端联动 -> “本地扣代币 → 通知后端，后端失败时回滚本地代币”。
*   **隐患分析**：如果用户网络有点卡，他在 RecordPage 的“兑换 AI 解析”按钮上**快速连点了三下**。
    *   你的前端会瞬间发起 3 个并发协程。这 3 个协程同时读取当前的本地代币余额，发现都够扣，于是它们同时向后端发送 3 次 `/redeem-tokens` 请求。
    *   后端大概率会给这个用户加上 3 次 AI 配额。虽然本地可能会通过回滚机制扣减 3 次代币，但如果本地数据库写入加锁处理不当，极易导致“用 1 次代币的钱，刷出 3 次配额”的并发漏洞。
*   **修复建议**：
    *   在 `CheckInViewModel` 或 `AiExpenseParser` 中引入一个 `isLoading` 或 `isRedeeming` 的 `StateFlow`。
    *   只要有兑换请求正在飞行（in-flight），UI 层就把兑换按钮 `enabled = false`。
    *   利用 Kotlin 协程的 `Mutex`（互斥锁）或后端的事务（Transaction）来保证防抖和原子性。

### 四、 财务逻辑陷阱：多币种股票池的“汇率幻觉”
*   **记录出处**：多货币汇率换算 -> “股票页：单个持仓保持原币种显示，总资产/总盈亏换算成主货币加总... currency 随刷新写入”。
*   **隐患分析**：在金融 App 中，“多币种持仓汇总”是极其复杂的。假设你的主货币是 **人民币(CNY)**，你花 **$100 美元** 买了一股苹果公司。
    *   买入时汇率：1 USD = 7.0 CNY（成本 700 元）
    *   一个月后，苹果股票价格完全没动，还是 **$100 美元**，但美元升值了，汇率变成了 1 USD = 7.2 CNY。
    *   如果按照你目前的逻辑简单加总实时汇率，总资产会变成 720 元，系统会显示你**“浮盈 20 元”**。但这其实是**汇率收益**，而不是**股票收益**。如果你的系统没有分别记录买入时的原始汇率成本，用户看到盈亏数据会觉得莫名其妙。
*   **修复建议**：
    *   像记账一样，股票的 `stocks` 表必须包含 **`costBasisExchangeRate` (买入时的锁定汇率)**。
    *   在分析持仓时，明确分离“资本利得（股价变动引起的）”和“汇率损益（汇率变动引起的）”。如果太复杂，退一步的做法是：**总盈亏只在同币种内计算，绝不要把不同币种的盈亏换算后强行加起来**。

### 五、 Android 前端实现：略微“复古”的语言切换方案
*   **记录出处**：多语言实现 -> “AppCompatDelegate.setApplicationLocales() 依赖 AppCompatActivity... MainActivity 必须继承 AppCompatActivity... Activity 自动重建”。
*   **隐患分析**：这其实不是一个大错，而是一个“技术债”。在纯 Jetpack Compose (Material 3) 的现代开发中，强制继承 `AppCompatActivity` 并依靠 `Activity` 重建来切换语言，是基于老版本 XML View 时代的妥协方案。Activity 重建会导致屏幕闪烁，且如果没有正确实现 `SavedStateHandle`，用户填了一半的表单数据会全部丢失。
*   **进阶方向**：在 Compose 中，可以通过提供自定义的 `Locale` 给底层的 `CompositionLocalProvider`，实现**热切换（Hot-swap）**：不重建 Activity，UI 瞬间丝滑变换语言。你可以让 Copilot 查阅 Android 13/14 推荐的按应用设定语言（Per-App Language Preferences）最佳实践，这会让你的前端技术栈看起来更 Modern。
